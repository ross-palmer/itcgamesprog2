A quick guide to using Github for team projects

In brief, Git is a source code management system, while Github is a cloud based hosting repository for Git projects. You are going to use Github to host your Joint project.

1. Creating a new repository

1.1 In Windows, make a new folder for your project (e.g. G:\JointProject).

1.2 Open the command line tool Git Bash - this will enable you execute common git commands.

1.3 Type the command cd G:\JointProject and press return. This means change directory (cd) to your JointProject folder.

1.4 For the team lead only - open this URL:

https://classroom.github.com/g/6F84jY-6

and await instructions from me on naming the repository.

2. Checking out a repository

2.1 Create a working copy of a local repository from a remote repository by running the command:

git clone https://github.com/ITCGamesProg2/jp-TeamName.git

2.2. You should now find a directory in your local file system with the same name as the above repository (e.g. jp-TeamName in this case). Change into this directory using the cd command, e.g. cd jp-TeamName

2.3 Download my README.MD from here. Copy it into your local repository directory. MD stands for MarkDown, which is a simple markup language that can be easily converted to HTML. Open the file and fill in the project name and project description only - do not remove any of the # symbols as they represent headings.

2.4 Add the readme to your remote repository with these commands:

git add README.MD
git commit -m "Added README file for project"
git push origin master

3. Creating a branch

You should always start by creating a new branch for the feature you are going to work on. This is particularly important for team projects. Branches are used to develop features that are isolated from each other. The master branch should always contain a clean working build of the entire project. Use other branches for development and merge them back to the master branch upon completion.

3.1 To create a new branch in your local repository called feature_x and to switch to it use:

git checkout -b feature_x

Note the use of the -b switch here.

3.2 When you have finished editing code, you need to commit your work back to the remote repository. If you have modified any existing files (most likely) OR added any new files, you need to stage them for the first commit:

git add .

This adds all files in the current directory and any subdirectory to the staging area, thus preparing them to be included in the next git commit.

3.3. Commit the files that you've staged in your local repository:

git commit -m "First commit"

'First commit'` is a commit string - this should generally summarise the changes you have made to the code base, so this will obviously change between commits.

3.4 Push the branch to the remote repository:

git push origin feature_x

3.5 Checking out another branch

Say another team member has pushed a new feature to a new branch called feature_y on the remote repository. If you want to checkout this new branch to your local repository, first switch to the master branch:

git checkout master

then pull the feature_y branch you are interested in:

git pull origin feature_y

Finally, you can checkout the new branch with:

git checkout feature_y

Note, that no -b switch is used here as we are not creating a new branch.

4. Merging your feature branch with the remote master

When you have finished working on your feature, you will need to merge your work with the master branch on the remote repository.

In team projects, merge conflicts can easily happen. This means when two identical branches have changed the same part of the same file. For example, if two people changed the same lines in that same file, or if one person decided to delete it while the other person decided to modify it, Git simply cannot know what is correct.

So, once your feature is finished and you are ready to merge your branch with the master branch. Perform steps 3.2 and 3.3 above (add/commit) then issue this command:

git pull origin master --rebase

The first part of this command means merge the remote master branch into the current branch. The --rebase means Git will rewind (undo) all of your local commits, pull down the remote commits then replay your local commits on top of the newly pulled remote commits. If any conflicts arise that git can’t handle you will be given the opportunity to manually merge the commits. Merge conflicts are highlighted by git like this:

<<<<<<< HEAD
x++; // This line is present on the remote repository
=======
y++; // This line is present on the local repository 
>>>>>>> Changed local file

So, on the remote repo, let's say a source file called Game.cpp has a line x++; while on your local repo, the same source file on the same line has y++; Let's pretend the line y++; is correct. To fix the above merge conflict, we remove everything except the correct line, i.e.:

y++; // This line is present on the local repository

Once you have fixed all the merge conflicts in that file, add the file to your staging area with:

git add Game.cpp

then give the command:

git rebase --continue

to carry on replaying your local commits, repeating the above steps to resolve all merge conflicts. When you are finished:

Switch to master branch: git checkout master Merge the feature branch with the master branch: git merge feature-name Push to the remote master branch: git push origin master

5. If it all goes wrong...

If you instead want to drop all your local changes and commits, fetch the latest history from the server and point your local master branch at it like this:

git fetch origin

git reset --hard origin/master

6. Deleting files from the remote repository

If you want to delete files from the remote repository, delete them from your local repository first and then give these commands:

git add -u

The -u means we are telling github to untrack the deleted files. Then commit and push:

git commit -m "Deleted branch files"

git push origin BranchName
