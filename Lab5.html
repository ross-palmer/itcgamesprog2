5.2 Generating the walls

The first step is to get the level data for the Walls from our Level object - remember, the Game class has a member variable m_level which represents our level data. Note we don't need to make any changes to our LevelLoader as it will happily load up the additional walls we have defined in the yaml file. Add a new private member function to Game.h as follows:

void generateWalls();

We will need to create a sf::Sprite for each wall, and because there are multiple walls, we need to store the sprites in a container. Add the following new member variable:

// A container for the Wall sprites
std::vector<std::unique_ptr<sf::Sprite>> m_wallSprites;

This defines a std::vector of type std::unique_ptr, where each unique_ptr will point to an instance of sf::Sprite. The vector of course will hold nothing yet, so we need to write the function generateWalls() next to fill it. In Game.cpp, define the new member function:

void Game::generateWalls()
{
    sf::IntRect wallRect(2, 129, 33, 23);
    // Create the Walls 
    for (ObstacleData const &obstacle : m_level.m_obstacles)
    {
        std::unique_ptr<sf::Sprite> sprite(new sf::Sprite());
        sprite->setTexture(m_texture);
        sprite->setTextureRect(wallRect);
        sprite->setOrigin(wallRect.width / 2.0, wallRect.height / 2.0);
        sprite->setPosition(obstacle.m_position);
        sprite->setRotation(obstacle.m_rotation);
        m_wallSprites.push_back(std::move(sprite));
    }
}

What is new here is how the sprites for each wall are added to the vector. We use dynamic memory allocation to create a sprite inside the loop:

std::unique_ptr<sf::Sprite> sprite(new sf::Sprite());

This line allocates a sprite on the heap, when is then pointed to by the sprite variable (a unique_ptr). These heap resources (i.e. the sprites we are creating) are known as managed objects. The sprite properties are then configured, before we use a std::move operation to move the sprites unique_ptr into the vector:

m_wallSprites.push_back(std::move(sprite));

Why is this move operation required? All STL containers copy in and copy out. So, if we were to to do this instead:

m_wallSprites.push_back(sprite);

It would mean copy the unique_ptr into the vector, and unique_ptr cannot be copied for obvious reasons!

In the Game constructor function, add a line to call the new member function:

generateWalls();

5.3 Collision detection between tank and walls

Now the walls are visible, the next step is to add collision detection between the tank and walls. Add the following new member function to Game.h:

bool checkTankWallCollision();

and in Game.cpp, add the implementation:

bool Game::checkTankWallCollision()
{
    for (std::unique_ptr<sf::Sprite> const & sprite : m_wallSprites)
    {
        if (CollisionDetector::collision(m_tank->getTurretSprite(), *sprite))
        {
            return true;
        }
    }
    return false;
}

Note: you will need to add a getter member function to the Tank class to return the sf::Sprite representing the turret, like this:

sf::Sprite const & getTurretSprite() const;
